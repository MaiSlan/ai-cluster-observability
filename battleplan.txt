Phase 1: Context-Aware Database Schema (Supabase / PostgreSQL)The foundation of a robust telemetry pipeline is how the data is structured. Instead of dumping flattened JSON logs into a single massive table, the database needs an architecture that maintains the parent-child relationships of the physical hardware. This prevents orphaned metrics and allows for deep-dive queries later.clusters Table: The top-level parent containing global metadata (e.g., cluster_id, region, status).nodes Table: The physical servers housing the hardware (e.g., node_id, cluster_id_fk, ip_address).gpus Table: The individual compute cards (e.g., gpu_id, node_id_fk, pci_bus_id).metrics_vllm Table: Time-series data tracking inference workloads. Columns must include gpu_id_fk, timestamp, vllm:time_to_first_token_seconds, vllm:time_per_output_token_seconds, and vllm:num_requests_waiting.  metrics_nccl Table: Time-series data tracking multi-GPU training communication. Columns should include gpu_id_fk, timestamp, bus_bandwidth_gbps, execution_time_us, and straggler_detected.Keeping this hierarchy intact ensures that writing complex SQL queries to isolate a single failing GPU inside a specific rack remains highly efficient.Phase 2: The Python Telemetry GeneratorInstead of relying on an AI-dependent middleware step to simulate data on the fly, this ingestion layer should be handled by a direct, deterministic Python script running on a local or cloud server configuration. This ensures a constant, lightweight stream of telemetry.The Core Logic: Write a Python script using psycopg2 or the supabase-py client that runs an infinite loop or triggers via a Cron job every minute, generating realistic hardware fluctuations.Simulating Inference (vLLM): Generate standard serving metrics. Keep vllm:time_to_first_token_seconds baseline around 0.2s, but occasionally inject spikes up to 2.0s precisely when the vllm:num_requests_waiting value increases.  Simulating Training (NCCL): Modern NVIDIA NCCL Inspector tools process communication performance data and export metrics directly into time-series databases like Prometheus. Your script will mimic this exact behavior by writing simulated bandwidth drops (e.g., from 400 GB/s down to 45 GB/s) directly into your PostgreSQL tables to represent sudden network congestion.  Phase 3: Creating SQL Views for the DashboardBefore building the frontend, the raw time-series data must be aggregated at the database level so the web application does not crash trying to download millions of rows.view_cluster_health_1m: Create a PostgreSQL view that calculates the moving average of bus_bandwidth_gbps and the maximum vllm:time_to_first_token_seconds per node over a rolling 60-second window.  view_straggler_alerts: Create a targeted view that only returns rows where a GPU was flagged with straggler_detected = TRUE in the last five minutes, providing an immediate endpoint for error logging.Phase 4: Frontend Development & VisualizationThe web interface needs to look like an enterprise-grade infrastructure command center.Framework: Use Next.js (App Router) and deploy the frontend on Vercel.Data Fetching: Utilize Next.js Server Components to securely fetch the aggregated SQL views from Supabase without exposing database credentials to the client.Charting Libraries: Implement Recharts or Tremor. Tremor is particularly well-suited for observability dashboards and handles dense time-series data cleanly.The Layout:Top KPIs: Bold metrics showing Total Requests Waiting, Average TTFT, and Active Cluster Bandwidth.Center Graph: A dual-axis line chart overlaying bus_bandwidth_gbps against execution_time_us. This visually proves an understanding of how network layer degradation causes compute latency.Bottom Grid: A status list of individual nodes. If a node's bandwidth drops below a set threshold, its status indicator turns red.Phase 5: The "Fault Injection" MechanismTo make the portfolio piece highly interactive during an interview, implement a manual mechanism to break the system on command.The Trigger: Add a "Simulate Network Degradation" button on the Next.js dashboard.The Action: Clicking this triggers a Server Action that updates a fault_status boolean flag in the Supabase clusters table.The Reaction: The Python script reads this flag on its next execution cycle. If it detects the fault is active, it forces the generated NCCL bandwidth metrics to plummet and the vLLM queue metrics to skyrocket. The frontend charts will react to this cascading failure in real-time, perfectly demonstrating a live cluster anomaly.

Tables & Columns
1. clusters (The top-level hierarchy)

id (UUID, Primary Key)

name (Text) - e.g., "eu-west-cluster-alpha"

fault_simulation_active (Boolean, Default: FALSE) - The toggle your Next.js button will hit.

created_at (Timestamptz)

2. nodes (The physical servers)

id (UUID, Primary Key)

cluster_id (UUID, Foreign Key -> clusters.id)

hostname (Text)

status (Text) - e.g., "online", "degraded"

3. gpus (The individual compute units)

id (UUID, Primary Key)

node_id (UUID, Foreign Key -> nodes.id)

model (Text) - e.g., "NVIDIA_A100_80GB"

pci_bus_id (Text)

4. metrics_vllm (Inference Telemetry - High Volume)

id (BigInt, Primary Key, Identity)

gpu_id (UUID, Foreign Key -> gpus.id)

timestamp (Timestamptz, Indexed for fast time-series graphing)

cache_usage_perc (Float) - How full the KV cache is (0.0 to 1.0)

requests_waiting (Integer) - The queue length

time_to_first_token_ms (Integer) - Latency metric

5. metrics_nccl (Networking Telemetry - High Volume)

id (BigInt, Primary Key, Identity)

gpu_id (UUID, Foreign Key -> gpus.id)

timestamp (Timestamptz, Indexed)

bus_bandwidth_gbps (Float) - Network speed

execution_time_us (Integer) - Microseconds to complete a sync

straggler_detected (Boolean) - Flags if this specific GPU is holding up the cluster

Row Level Security (RLS) Strategy
Because this dashboard will be public on Vercel for recruiters to see, you must lock down the data.

The Python Backend (Service Role):
Your Python script will use the SUPABASE_SERVICE_ROLE_KEY in its .env file. This key completely bypasses RLS. It can INSERT, UPDATE, and DELETE at will.

The Next.js Frontend (Anon Role):
The Next.js app will use the NEXT_PUBLIC_SUPABASE_ANON_KEY. We will set RLS policies on all tables so that the anon role can ONLY SELECT (read) the data.

Exception: We will create a specific PostgreSQL RPC (Remote Procedure Call) function called toggle_cluster_fault(). We will grant the anon role permission to execute this specific function so the frontend button can flip the fault_simulation_active boolean without giving the frontend raw UPDATE access to your tables.
